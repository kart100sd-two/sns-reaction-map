# Article Workflow

作成日: 2026-06-13

## 目的

論争・政策・事件・制度改革などを扱う記事を、毎回同じ品質で作成するための標準運用手順。

このワークフローは特定事例に依存しない。個別事例の内容、固有名詞、具体的な主張は、別途サンプル記事または記事下書きファイルに記載する。

## 基本原則

- 事実、報道上の主張、本人説明、本人否定、SNS上の推測、編集部分析を分ける。
- 賛成/反対だけでなく、擁護、批判、保留、中立を扱う。
- すべての重要な主張に根拠リンクを付ける。
- 根拠が弱い情報は「未確認」と表示する。
- SNS上の論調は、代表性や拡散規模を過大評価しない。
- 記事は断罪や煽りではなく、論点整理を目的にする。

## 全体フロー

| Step | 作業 | 成果物 |
| --- | --- | --- |
| 1 | テーマ選定 | 記事候補メモ |

## Step 1: テーマ選定

### 選定基準

| 基準 | 内容 |
| --- | --- |
| 公共性 | 社会、政治、政策、制度、生活に関係する |
| 論点性 | 賛否や意見の対立がある |
| 根拠性 | 報道、公式資料、本人発言など確認できる資料がある |
| 継続性 | 続報や反論が出る可能性がある |
| 安全性 | 個人情報、未成年、被害者保護のリスクを管理できる |

### 避けるテーマ

- 個人攻撃が中心の話題
- 根拠がSNSスクショだけの話題
- 被害者や未成年の特定につながる話題
- 誹謗中傷の再拡散になりやすい話題

## Step 2: ソース収集

### 集めるべきソース

| 種別 | 例 |
| --- | --- |
| 一次資料 | 公式発表、国会会議録、議事録、法案、統計、本人投稿 |
| 報道 | 新聞、テレビ、通信社、調査報道 |
| 当事者説明 | 会見、声明、SNS、公式サイト |
| 反論 | 本人否定、訂正、第三者反論 |
| 専門家 | 法律、制度、政策、現場専門家の解説 |
| SNS | 代表的な賛否投稿、拡散された投稿 |

### SNSの扱い

SNSは補助材料として扱う。全体世論の代表として扱わない。

記録する項目:

- URL
- 投稿者
- 投稿日時
- 立場
- いいね、リポスト、表示数
- 引用元
- 削除可能性

## Step 3: ソース台帳化

各ソースを構造化する。

| 項目 | 内容 |
| --- | --- |
| source_id | 一意のID |
| title | タイトル |
| url | URL |
| publisher | 発信元 |
| published_at | 公開日 |
| source_type | official / news / sns / expert / document |
| evidence_level | high / medium / low / unknown |
| summary | 100-200字の要約 |
| risks | 注意点、反論、未確認点 |

## Step 4: 情報分類

記事化する前に、情報を次のラベルへ分類する。

| ラベル | 意味 |
| --- | --- |
| confirmed_fact | 確認済み事実 |
| reported_claim | 報道上の主張 |
| official_statement | 公式説明 |
| personal_explanation | 本人説明 |
| denial | 本人・関係者の否定 |
| unverified_claim | 未確認情報 |
| sns_opinion | SNS上の意見 |
| editorial_analysis | 編集部分析 |

## Step 5: 争点抽出

争点は問いの形で作る。

悪い例:

- A氏は悪い
- B党は正しい

良い例:

- 当事者の認識はどこまで確認できるか
- 制度上の問題なのか、運用上の問題なのか
- 費用対効果はどれくらいあるのか
- 批判と誹謗中傷の境界はどこか
- どの資料が主張を支えているのか

## Step 6: 賛否整理

立場は最低3分類にする。

| 立場 | 内容 |
| --- | --- |
| defense | 擁護・肯定 |
| critical | 批判・反対 |
| neutral | 保留・中立 |

各意見カードには次を含める。

| 項目 | 内容 |
| --- | --- |
| claim | 主張 |
| basis | 根拠 |
| weakness | 弱点・反論可能性 |
| evidence_ids | 根拠ソースID |
| confidence | high / medium / low |

## Step 7: 比較整理

複数の人物、政党、制度案、事例がある場合は比較表を作る。

比較軸の例:

- 関与の種類
- 根拠の強さ
- 公式説明の有無
- 本人否定の有無
- 影響範囲
- 未確認点
- 暫定評価

## Step 8: 下書き作成

記事は次の順番を基本にする。

```markdown
# タイトル

最終更新: YYYY-MM-DD
ステータス: 調査中 / 更新中 / 概ね整理済み

## 30秒要約

## 確認済み・報道上の主張

## 未確認・要検証ポイント

## 争点マップ

## 擁護側の意見

## 批判側の意見

## 保留・中立の意見

## 関連事例・比較表

## 時系列

## 根拠リンク

## 編集部メモ
```

### バトル形式を使う場合

SNS上で賛否が強く分かれているテーマは、標準記事に加えて、または標準記事の代わりに「論点バトル形式」を使う。

目的は勝敗を煽ることではなく、賛否双方の主張、根拠、弱点、未確認点を同じ土俵に置くこと。

基本構成:

```markdown
# {テーマ名} 論点バトル

最終更新: YYYY-MM-DD
ステータス: 調査中 / 更新中 / 概ね整理済み
形式: 擁護派 vs 批判派

## まず3行で

## 登場する主な論点

# Round 1: {論点}

## 批判派

### 根拠

## 擁護派

### 根拠

## 現時点の整理

根拠の強さ:

| 立場 | 強さ |
| --- | --- |
| 批判派 | ★★★☆☆ |
| 擁護派 | ★★★☆☆ |

# 総合整理

## 批判派が強いところ

## 擁護派が強いところ

## 今回の本丸

## 暫定結論

## 根拠リンク

## 編集部メモ
```

### バトル形式のルール

- 勝敗ではなく、根拠の強さを見せる。
- SNSの攻撃的表現はそのまま転載せず、主張へ変換する。
- 各ラウンドは「問い」の形にする。
- 未確認情報は必ず未確認と表示する。
- 本人否定や反論がある場合は同じラウンド内で扱う。
- 星評価は世論比率ではなく、現時点で確認できる根拠の強さを示す。

## Step 9: 人間レビュー

公開前に必ず人間が確認する。

### チェックリスト

- 事実と意見が分離されているか
- 報道上の主張を確定事実として書いていないか
- 本人否定や反論を掲載しているか
- 未確認情報にラベルが付いているか
- 根拠URLが付いているか
- SNS論調の代表性を過大評価していないか
- 見出しが煽りすぎていないか
- 個人攻撃や差別表現を増幅していないか
- 引用が長すぎないか
- 訂正可能な構造になっているか

## Step 10: Web公開

### 推奨ページ構成

- タイトル
- 30秒要約
- ステータス
- 最終更新日
- 主要ラベル
- 確認済み/未確認
- 争点マップ
- 賛否カード
- 比較表
- 時系列
- 根拠リンク
- 編集履歴

### ステータス

| ステータス | 意味 |
| --- | --- |
| investigating | 調査中 |
| updating | 続報更新中 |
| stable | 概ね整理済み |
| corrected | 訂正あり |

## Step 11: Newsletter化

### 無料版

- 30秒要約
- 主要論点
- 代表的な賛否
- 根拠リンクの一部
- Webページへの導線

### 有料版

- 詳細な時系列
- 立場別の詳細表
- 比較表
- 根拠リンク全文
- 未確認点一覧
- 今後見るべきポイント

## Step 12: 更新管理

新情報が出たら、本文を書き換えるだけでなく更新履歴を残す。

| 項目 | 内容 |
| --- | --- |
| updated_at | 更新日 |
| changed_section | 更新箇所 |
| reason | 更新理由 |
| source_id | 根拠ソース |
| editor | 編集者 |

## AIに任せる作業

- URL本文の要約
- 事実/主張/否定/未確認の分類
- 論点候補の抽出
- 賛否カードの下書き
- 比較表の下書き
- 時系列の下書き

## 人間が必ず確認する作業

- 根拠の妥当性
- 名誉毀損リスク
- 個人情報リスク
- 政治的偏り
- 本人否定や反論の有無
- 見出しの妥当性
- 公開可否
